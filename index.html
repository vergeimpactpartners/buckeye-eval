How this platform is structured, what each page is for, and how to use it.
What is VIPIlluminate?

VIPIlluminate is the evaluation workspace for the Buckeye EcoVillage Developmental Evaluation, built and managed by Verge Impact Partners. It is a single HTML file with three distinct access modes, each serving a different audience and purpose.

Three modes, three audiences
Evaluator
The full workspace for the evaluation team. Includes Project Planning, Village Experience, data collection views, sensemaking, and fidelity tracking. Controls what residents see. Only accessible to VIP team members.
EHW
A shared page for EVB partners and EHW team members. Shows evaluation aims, learning questions, and sprint progress. Read-only — no editing or publishing.
Village
A resident-facing record of the evaluation in progress. Shows the meeting timeline, activity records, and emerging themes. Content only appears here when explicitly published. Always opens to the default activity set by the evaluator.
Three URLs, three audiences
Evaluator  —  vergeimpactpartners.github.io/buckeye-eval/
Full workspace. All views available. Village Experience and publish controls visible.
EHW  —  vergeimpactpartners.github.io/buckeye-eval/?mode=ehw
Shared view for EVB partners. Shows published learning content. Read-only.
Village  —  vergeimpactpartners.github.io/buckeye-eval/?mode=village
Resident-facing page. No evaluator controls visible. Opens to the default activity automatically.
Evaluator navigation
Project Planning Sprint timeline, phase steps, and process documentation.
Village Experience Control panel for the Village page. Set the meeting timeline, choose the default activity, control which activities are visible to residents, publish/unpublish emerging themes, and use the Activity Builder to create new content with AI assistance.
Data Collection Overview + one page per Photovoice activity. Pages are generated automatically as Photovoice activities are created.
Sensemaking Sensemaking (AI chat), Themes, Fidelity tracker.
Platform Guide This page.
Village Experience — control panel

All controls for the resident-facing page live here. Never need to go to the Village page itself to manage it.

Meeting Timeline — Set and advance the current meeting. Edit and advance using the ✏ Edit button on the Up Next meeting.
Activities table — Three columns per activity: Default (which activity opens first for residents), Visibility (Published/Hidden), Themes (Unpublished/✓ Published). Changes save to Drive immediately.
Activity Builder — Four modes: (1) New BAR/AAR — create a full new activity with name, framing question (with bold/italic/underline formatting), optional subcategories, and BAR content filled in before publishing; (2) Create new data submission — create a Photovoice-style submission activity selecting type (photo, video, written story, audio); (3) Fill themes box — manually add themes with title and description to any activity, then publish to Village; (4) Edit existing activity — select any activity to pre-populate all fields including BAR rows and AAR questions (Actual results, Caused results, Sustain/improve) per subcategory. All structural changes auto-commit to GitHub.
Activity types
BAR / AAR — blue header
Data collection activities with a Before Action Review and After Action Review structure. Four sections: framing question, role pills, BAR columns (horizontally scrollable), action bubble, AAR, and emerging themes panel.
Photovoice — amber header
Resident submission activities. Residents submit photos, videos, written stories, or audio recordings. Each submission includes a theme tag and description. Submissions appear in a horizontal gallery and are saved to Google Drive. Evaluator can add emerging themes to these activities as well.
Publish system
Activity visibility — Toggle Published/Hidden per activity in the Village Experience table. Hidden activities are invisible to residents on the Village URL. In evaluator Village mode they appear dimmed with a "Publish to Village" button.
Emerging themes — Each activity has a magenta themes panel. Toggle Unpublished/Published in the Village Experience table, or use the "Publish to Village" button on the panel itself. Unpublished themes are hidden from residents; visible to evaluators at 50% opacity.
Default activity — The activity residents land on when opening the Village link. Set in the Village Experience table. Persists to Drive.
Submissions open/closed — For Photovoice activities, an amber "Open/Closed" button in the Village Experience table controls whether residents can submit. When closed, the submission form is hidden and replaced with a notice; the gallery remains visible. State persists to Drive.
Color system
Blue — process, navigation, BAR/AAR activity headers
Magenta — learning, themes, sensemaking
Amber — Photovoice activities, up next
Midnight — sensemaking, coding, AI
Green — completion, published states
Blue-pale — structure, framing sections
Drive sync

Data syncs to Google Drive via César's vergeimpactpartners.com account through a Google Apps Script. The following are synced across all evaluator devices:

Publish state — activity visibility, default activity, theme publish status, submissions open/closed
BAR row content — "What will make us successful?" text per subcategory, restored on every page load
AAR content — Actual results, Caused results, Sustain/improve per activity, restores and undims AAR table on load
Theme card content — theme names and notes for each activity, restored on every page load
Inline text edits — any text edited directly on the village page (activity titles, framing questions, descriptions, etc.)
Photovoice submissions — saved to Google Sheet with media files in Drive; gallery restores on every page load
Generated activities — new activities created via the Activity Builder, restored on every page load
Sprint state — which sprint is active, task completions, phase notes, content cards
Fidelity tracker — sprint assessments and trend data across sprints
● Drive — Data syncing to Google Drive. All evaluators see the same state.
○ Local retry — Drive temporarily unavailable. Data saved locally. Click retry to reconnect.
How to update the platform

The platform is a single HTML file deployed via GitHub Pages (managed by César under the vergeimpactpartners org). There are two types of changes — automatic and manual.

Automatic — no action needed
Structural changes made inside the platform auto-commit to GitHub 2 seconds after saving. This includes: creating new activities via the Activity Builder, editing BAR content, filling AAR, and inline text edits. Drive state (themes, visibility, submissions, sprint progress) saves to Drive only — not GitHub.
Manual — for code changes in Claude
When you need to make structural code changes (new features, layout changes, etc.), bring the current file into a Claude session. Since the auto-commit keeps GitHub current, always pull the latest file from GitHub — not a locally saved copy.
1Go to github.com → vergeimpactpartners → buckeye-eval → index.html
2Click Raw → Select all (Cmd+A) → Copy
3Open Claude → start a new chat → paste the file and describe the changes needed
4Download the updated file from Claude
5Go back to GitHub → click the pencil icon → Select all (Cmd+A) → Paste (Cmd+V) → Commit changes
6GitHub Pages auto-deploys within 60 seconds
Tip — continuity across sessions: Claude remembers the platform's history and structure. If starting a new chat, mention this is the VIPIlluminate platform for Buckeye EcoVillage DE and share the current file — Claude will pick up right where you left off.
Inline text editing

In the evaluator link, any text on the Village page can be edited directly without coming back to Claude. Edits are only visible to evaluators — residents never see the edit controls.

Switch to Village mode (using the Evaluator / EHW / Village switcher in the top bar). Hover over any text — a dashed blue outline and ✏️ appears. Click to open an edit popover with the current text pre-filled.
Editable text includes: activity titles, framing question preambles, framing questions, subcategory headers and names, action labels, BAR row content, and Photovoice activity descriptions.
How edits save: Each edit saves immediately to Drive under a key like ie-act2-framing-pre. It also triggers an auto-commit to GitHub so the edited text is baked into the file for future Claude sessions. On every page load, all saved text edits are restored from Drive.
Sensemaking

The Sensemaking section is the AI thinking partner for the EVB evaluation. It is the hub for interpreting data, surfacing patterns, and generating questions — not a reporting tool.

Sensemaking (AI chat) — A persistent conversation with Claude, using the full EVB DE system prompt. Load activity data directly into the chat by selecting one or more activities and choosing which data types to include (Activity content, Themes, Submissions). Quick prompts in the right panel are context-aware — if an activity is selected, they generate focused prompts referencing that activity's data.
Themes — A live view of all published theme cards across activities, organized by activity. Updates when you publish themes from the Activity Builder, or click ✦ Refresh themes. A ✦ Generate synthesis button runs AI analysis across all published themes to surface connections and tensions.
Fidelity tracker — A structured self-assessment of the evaluation's fidelity to EVB's principles and the developmental evaluation approach. Completed by the evaluator team at key sprint milestones.
Theme updates: Theme data is Drive-only — it updates every time you publish themes and does not require a GitHub commit. The Themes page reads from Drive on load and always reflects the current published state.
Sprint launching

The Project Planning page supports multiple learning loop sprints. Sprint 2 is pre-built but inactive until launched. Additional sprints can be added dynamically for projects with more than two learning loops.

Launching Sprint 2 — In the Project Planning page, Sprint 2 shows as ⏳ Upcoming with a "Ready to begin Sprint 2?" prompt. Clicking Launch Sprint 2 → populates Sprint 2 with the same phase and task structure as Sprint 1, switches the progress bar to Sprint 2, and saves the launch state to Drive + GitHub.
Additional sprints — A "Launch next loop →" button appears at the bottom of each sprint's Step 4 once that sprint is active. This generates a new sprint dynamically with the same structure. Works for any number of sprints — designed to support future projects beyond EVB's two-sprint structure.
Progress bar — Shows only the active sprint's steps, with a "Sprint X of Y" indicator when more than one sprint exists. Clicking a step in the bar navigates directly to that step in Project Planning.
Photovoice gallery
Submissions appear immediately in the gallery after submit. On page reload, submissions are restored from the Google Sheet via the GAS backend.
Clicking any photo opens it full-screen in a lightbox overlay. Click anywhere, press Escape, or click ✕ to close. Does not open a new tab.
To delete a submission: remove the row from the photovoice_submissions spreadsheet in the VIPIlluminate Drive folder. The media file in the photovoice/ subfolder must be deleted separately. There is no in-platform delete button currently.
Use the Open/Closed toggle in the Village Experience table to close submissions without hiding the gallery.
Sprint progress bar
The bar at the top of Project Planning shows four steps (Set up, Learning Loop, Adapt, Close). It updates automatically as you check off tasks.
Gray — no tasks checked. Blue — some tasks checked (in progress). Green ✓ — all tasks complete. Multiple steps can be blue simultaneously.
AI features — current status

The platform includes AI features for BAR synthesis, sensemaking, and theme analysis. These currently do not work on the deployed GitHub Pages URL.

Why they don't work on the deployed URL
The features call the Anthropic API directly from the browser. This works inside Claude.ai's environment but is blocked from external sites like GitHub Pages — likely due to CORS restrictions or authentication requirements specific to Anthropic's API.
What would fix it — Routing AI calls through the GAS backend (a proxy function with an Anthropic API key in Script Properties). Requires an Anthropic API account with billing. The code changes are ready to build when this is prioritized.
Current workaround — Do AI synthesis directly in Claude.ai using the EVB DE system prompt. Paste source material, receive BAR content, enter it manually via Activity Builder → Edit existing activity.

Framing Question

              
What would it take to successfully hold the EVB Way while planning and carrying out the EVB kickoff?

              

                
There are four ways of working that came up during our evaluation kickoff. They are:

                

                  

                    
Residents set priorities:

                    
residents establish the priorities for EVB, partners and organizations need to align with them

                  

                  

                    
Space to disagree:

                    
hold space for people to disagree authentically

                  

                  

                    
Different outcomes:

                    
be open to different outcomes and measures of success than the ones originally conceived

                  

                  

                    
Values:

                    
establish formal structure with agreed upon values, commitments, roles, power, and permission

                  

                

              

            

            
            

              

                
                

                  

                    
What will make us successful?

                  

                  

                    

                      
Overall

                      
We need to make a strategic outreach plan for the entire summer, using unique outreach strategies and meeting people where they are already gathering, to ensure we have a great turnout for the kickoff and that Buckeye residents understand the context of EVB and feel welcomed and empowered to share their input.

                    

                    

                      
Residents set priorities

                      
Every outreach touchpoint with the community is an opportunity to share about the EVB process and have the community share their own goals. We need to innovate the way we do this outreach, focusing beyond door knocking and instead build a strategy for neighboring in unique ways that invite in those who traditionally don't participate (like youth).

                    

                    

                      
Space to disagree

                      
We acknowledge that disagreement is a common part of any process, but the way we respond to disagreement is important. We need to establish a process that makes sure that everyone involved feels heard and that their contribution is valued. Our response needs to remind everyone that we are on the same team and have shared goals.

                    

                    

                      
Different outcomes

                      
To get to different outcomes (high turnout and consistent engagement) we need to meet and invite the community at places and times where they are already gathering and where trust is established. We hope this will help us recruit people who are excited and ready to participate and take on the EVB work.

                    

                    

                      
Values

                      
We need to remind ourselves and the community at large what our shared values are and why we have set them to be that. The way we engage with each other and the community need to bring these values to life.

What will make us successful?

                  

                  

                    

                      
Setting Up Green Spaces & the Pod

                      
This work requires a real team — one person cannot carry it alone. We need volunteers who are motivated by purposeful causes, youth involvement especially in summer, and partnerships with organizations already doing youth programming. Land access through the city or land banks needs to be navigated early. The community needs to feel ownership over what gets built. We have a group that is aligned and wants to see this through — that's a real asset.

                    

                    

                      
Storytelling

                      
Success means people feel proud of their neighborhood and good about being asked to share — not just that stories get collected. Capturing stories in multiple formats matters because literacy and camera shyness are real barriers; meeting people where they are is part of the method. Leading with good faith and good intentions — especially when navigating roles across partners and spaces — is what keeps the work grounded when collaboration gets complicated. And the proof of concept is already there: we know this creates something that lands in people's heart space. Build on that feeling.

                    

                    

                      
Establishing Resident-Derived Priorities

                      
Success means residents can explain the EcoVillage — what eco, village, and sustainability mean — in their own words, and feel ready to invite someone else in. We're building from lessons already learned: different approaches work for different people and ages, and we need ambassadors who can hold space for disagreement without losing the relationship. Planning ahead for known barriers — childcare, transportation, work, conflict — is part of the work, not an afterthought.
